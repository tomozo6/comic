# 漫画閲覧サイト v1 — プロダクトバックログ

`issues/epic.md` を実現するための、順序付きユーザーストーリーです。1 ストーリーは
「実装・レビュー・検証までを一つの PR で終えられる」大きさを目安にしています。

## 共通の完了条件（Definition of Done）

- 受け入れ条件をローカルで確認済みである。
- 追加・変更した Go コードには自動テストがある（UI / Terraform は、確認手順を PR に記す）。
- 秘密情報、実ユーザーの個人情報、画像の実体をリポジトリへ含めない。
- README または運用手順を、利用者に影響する変更に合わせて更新する。

## 優先順位付きバックログ

### US-001: 秘密情報を安全に管理する

**ユーザーストーリー**: 運用者として、OAuth クライアントシークレットを安全な場所で管理したい。誤ってリポジトリから漏えいさせずに認証基盤を運用するためである。

**受け入れ条件**

- OAuth クライアントシークレットが Terraform ソース、サンプル、コミット対象ファイルに含まれない。
- Secret Manager または `terraform.tfvars`（gitignore 対象）から値を渡せる。
- 既に露出したシークレットは失効・再発行する手順が文書化されている。
- `terraform validate` が通る。

### US-002: 読み取り専用カタログを再現可能に生成する

**ユーザーストーリー**: 運用者として、作品・章・ページと画像オブジェクトキーをソース管理し、同じカタログ DB を再現可能に生成したい。

**受け入れ条件**

- SQLite の DDL と初期データをリポジトリで管理する。
- `catalog.db` をビルド時に生成し、実行イメージへ同梱できる。
- 作品、作者、巻、章、ページ、GCS object key を表現できる。
- 不連続なページ番号、重複した object key などを検出する検証コマンドまたはテストがある。

### US-003: ログイン状態を API で確認する

**ユーザーストーリー**: 家族の閲覧者として、Google でログインし、自分が閲覧できる状態だと確認したい。

**受け入れ条件**

- `GET /api/me` は有効な Firebase ID token で uid・表示名・メールアドレスを返す。
- token が無い、不正、期限切れの場合は JSON の `401 Unauthorized` を返す。
- Firebase プロジェクト ID は環境変数で設定でき、ローカルテストでは検証器を差し替えられる。
- ブラウザは Firebase ログイン後、ID token を Authorization ヘッダー付きで送信できる。

### US-004: ログイン済み利用者が作品一覧を見られる

**ユーザーストーリー**: 家族の閲覧者として、ログイン後に読める作品の一覧を見たい。読みたい漫画を選べるようにするためである。

**受け入れ条件**

- `GET /api/manga` は認証済みユーザーにカタログの作品一覧を返す。
- 未認証の呼び出しは `401` になる。
- 一覧には最低限、作品 ID、タイトル、作者、表紙（存在する場合）の情報がある。
- UI で読み込み中・空・エラーの状態を区別して表示する。

### US-005: 漫画画像を private bucket に安全に配置する

**ユーザーストーリー**: 運用者として、漫画画像を公開せずに保管し、閲覧サービスだけが読めるようにしたい。

**受け入れ条件**

- Terraform で専用 GCS bucket と Cloud Run 実行用サービスアカウントを作成する。
- bucket は public access prevention を有効にし、一般公開 IAM を持たない。
- 実行用サービスアカウントだけに、対象 bucket の object read 権限を付与する。
- 画像配置・アップロード・カタログ整合性確認の手順がある。

### US-006: 章を開くと一時的な画像 URL を取得できる

**ユーザーストーリー**: 家族の閲覧者として、選んだ章のページを安全に読みたい。画像 URL が恒久的に第三者へ共有されないようにするためである。

**受け入れ条件**

- `GET /api/manga/{mangaId}/chapters/{chapterId}` は認証済みユーザーにページ順の画像情報を返す。
- 各画像 URL は private GCS object への署名付き URL で、有効期限は 10 分である。
- 未認証、存在しない作品・章、カタログにない object key を適切に `401` / `404` で扱う。
- 署名 URL 発行処理は単体テストで有効期限と object key を検証する。

### US-007: スマートフォンで漫画を連続して読める

**ユーザーストーリー**: 家族の閲覧者として、作品から章を選び、スマートフォンでページを順に読みたい。

**受け入れ条件**

- 作品一覧から章を選択してリーダーを開ける。
- ページが順番どおりに表示され、狭い画面でも横スクロールなく読める。
- 署名 URL の期限切れ時は、章情報を再取得して復帰できる。
- ログアウト後は作品・章の画面を表示せずログイン画面へ戻る。

### US-008: Cloud Run へ一貫してデプロイできる

**ユーザーストーリー**: 運用者として、同じ手順でアプリケーションとカタログをステージング環境にデプロイしたい。

**受け入れ条件**

- Cloud Run、Artifact Registry、必要な API と IAM を Terraform で管理する。
- コンテナは Cloud Run の `PORT` 環境変数で待ち受ける。
- `/`、`/api/me`、作品 API のステージング確認手順がある。
- `terraform plan` が意図しない変更なしに安定する。

## 推奨する進行順

1. **US-001** — まずシークレットを安全にする。
2. **US-002** — 以降の API/UI の共通データ基盤を作る。
3. **US-003 → US-004** — 認証された利用者が作品を選べる最初の縦切り。
4. **US-005 → US-006 → US-007** — private 画像を安全に読む体験を完成させる。
5. **US-008** — ステージングへ反復可能に届ける。

## 最初のイテレーション

最初のイテレーションでは **US-001** のみを着手対象にします。外部での OAuth
シークレット再発行が必要なため、再発行値を受け取る場所（Secret Manager か
ローカルの `terraform.tfvars`）を決めてから実装します。完了後は US-002 を次の
作業単位として着手します。
