# v1 アーキテクチャ

## 方針

- HTML 配信と API は Cloud Run の Go アプリケーション1つに同居させる。
- ビルド時には作品ごとの YAML から読み取り専用 SQLite を作成してコンテナイメージに同梱する。Cloud Run はリクエストごとに SQLite を参照して API 応答を生成して返す。
- 認証は Firebase Auth の Google ログインだけを使う。
- 漫画画像は private GCS bucket に置き、アプリケーションが短命の署名付き URL を返す。
- ログイン済みの家族全員が、すべての作品を読める。
- カタログを更新するときは SQLite を作り直し、イメージを再ビルドしてデプロイする。

## 構成

```text
Browser
  ├─ Firebase Auth で Google ログイン
  ├─ Cloud Run から HTML / CSS / JavaScript を取得
  ├─ Cloud Run API から認証済みのページ情報を取得
  └─ 署名付き URL を使って GCS から画像を表示

Cloud Run (Go)
  ├─ SQLite を参照して HTML を生成
  ├─ Firebase ID token を検証
  ├─ SQLite catalog を参照
  └─ GCS 画像の署名付き URL を発行

GCS
  └─ private bucket に漫画画像を保存

Build
  ├─ 作品カタログの SQLite を生成
  └─ SQLite と Go アプリケーションをコンテナイメージに同梱
```

## API

| エンドポイント | 用途 |
| --- | --- |
| `GET /api/me` | Firebase ID token が有効か確認する |
| `GET /api/manga/{mangaId}` | 指定した作品の巻一覧を返す |
| `GET /api/manga/{mangaId}/volumes/{volumeId}` | 指定した巻のページ一覧と短命の画像 URL を返す |

作品一覧・巻一覧は API が SQLite から取得する。漫画画像の URL は private GCS bucket を直接公開しないため、API からのみ取得する。

API は Authorization ヘッダーの Firebase ID token を必須とし、無効な token には `401 Unauthorized` を返す。
