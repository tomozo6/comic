# ADR 0007: SQLite カタログは作品と巻を保持する

- 状態: 採用
- 日付: 2026-07-19

## 背景

現在の画面は、作品一覧、作品ごとの巻一覧、巻のページ列を表示する。カタログの編集元は
YAML とし、アプリケーションは SQLite を参照するように変更する予定である。

## 決定

- カタログは `mangas` と `volumes` の2表で保持する。章・話・ページのテーブルは作らない。
- 作者は初期版では作品ごとに1名とし、`mangas.author_name` に保持する。
- 画像そのものは SQLite に格納せず、`cover_object_key` に表紙画像の参照先を保持する。
  ページ画像の参照先は、全巻共通の命名規則から復元する。
- `volumes` は `page_count` と `page_extension` を持つ。同一巻のページ画像はすべて同じ
  拡張子とする。
- 巻には表示用の `number` とは別に、URL と画像キーに使う不変の `id` を持たせる。
- YAML の編集元は `catalog/mangas/{manga_id}.yaml` とし、作品ごとに1ファイルで管理する。
- YAML から SQLite を生成する処理は1つだけ実装する。ローカル開発ではアプリ起動時に
  呼び出し、本番では Docker ビルド時に同じ処理を呼び出す。
- 外部に公開する ID は YAML で指定する安定した文字列 ID とする。

## 画面との対応

| 画面 | 参照する表 | 必要な項目 |
| --- | --- | --- |
| 作品一覧 | `mangas` | 作品 ID、タイトル、作者名、表紙参照先 |
| 巻一覧 | `mangas`、`volumes` | 作品情報、巻 ID、巻番号、巻タイトル、巻表紙参照先 |
| 漫画リーダー | `volumes` | 巻情報、ページ数、ページ画像を復元する規則 |

## 制約

- `volumes` は `UNIQUE(manga_id, number)` とする。
- 表紙は任意とする。未指定なら画面はプレースホルダーを表示する。
- リーダーは `page_count` と共通の命名規則から、1 から `page_count` までの画像キーを
  昇順で生成する。画像キーの命名規則を逸脱する巻は扱わない。
- ページ画像のキーは
  `manga/{manga_id}/{volume_id}/{page:03d}.{page_extension}` に統一する。
- 本番アプリケーションは生成済みの SQLite を読み取り専用で開く。SQLite を実行時の
  永続ストレージとして扱わない。
